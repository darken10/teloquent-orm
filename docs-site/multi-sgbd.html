Multi-SGBD

Le même code applicatif fonctionne sur SQLite, MySQL et PostgreSQL. Une couche Grammar par dialecte gère les différences de SQL.

Différences gérées automatiquement

AspectSQLiteMySQLPostgreSQL
Quoting des identifiants"col"`col`"col"
Placeholders??$1, $2
Clé générée à l'insertlastInsertRowidinsertIdRETURNING
UpsertON CONFLICTON DUPLICATE KEY UPDATEON CONFLICT
Types (boolean, json…)integer / texttinyint / jsonboolean / jsonb
Format de dateISOYYYY-MM-DD HH:MM:SSISO

Vous écrivez le même code ; Teloquent produit le SQL adapté à la connexion active.

Choisir le dialecte

await ConnectionManager.addConnection({ driver: "sqlite", database: "app.sqlite" });
// ou "mysql" / "pgsql" — le reste du code est identique

Tester sur plusieurs SGBD

Le dépôt fournit un docker-compose.yml (Postgres + MySQL) et des scripts pour rejouer la suite de tests sur chaque dialecte.

npm run db:up        # démarre Postgres + MySQL (Docker)
npm run test:sqlite  # tests sur SQLite (par défaut)
npm run test:pg      # tests sur PostgreSQL
npm run test:mysql   # tests sur MySQL
npm run db:down      # arrête et nettoie

Paramétrer par variables d'environnement

TELOQUENT_DIALECT=pgsql \
PG_HOST=127.0.0.1 PG_PORT=5433 PG_USER=teloquent PG_PASSWORD=teloquent PG_DATABASE=teloquent \
npm run test:pg

Intégration continue

Un workflow GitHub Actions exécute automatiquement la suite sur les trois SGBD (services Docker Postgres et MySQL), garantissant la compatibilité multi-dialecte à chaque commit.

Bonne pratique : développez sur SQLite (rapide, zéro setup) puis validez sur le SGBD de production via test:pg / test:mysql avant de livrer.
